### [CRITICAL] (wal-journal-crash) Bundle import file-swap leaves the previous archive's hot -wal/-shm, replayed into the freshly imported canonical archive

- **Trigger:** crash/power-loss before import (hot -wal left on disk) followed by mode-toggle/import
- **How:** apply_import (migration.rs:470-629) swaps DBs at FILE granularity: movable_files (lines 557-561) renames only `archive/history-vault.sqlite` to `.bak-<ts>` and then moves the staged bundle DB into place (lines 562-578). It never relocates `history-vault.sqlite-wal`/`-shm`. The comment at lines 550-552 asserts that because the caller stopped workers there are no live WAL files — but 'workers stopped' only means no OPEN handle; a prior crash / force-quit / power loss leaves a hot `-wal` (committed-but-uncheckpointed frames) on disk (the documented prior-incident condition). After the swap, the bundle's self-contained `history-vault.sqlite` (produced via sqlcipher_export, no WAL) now sits next to the OLD database's leftover `-wal`/`-shm`. Step 5 then calls open_archive_for_migration (migration.rs:631-640) -> open_archive_connection -> run_migrations (lines 614-615), which opens in WAL mode; SQLite finds the foreign `-wal`, accepts its header (same page size / same at-rest mode is the common case) and replays the OLD db's frames (with OLD page numbers) into the NEW imported DB, overwriting pages 1..N. For a same-mode plaintext pair this is structural corruption; for an encrypted leftover replayed over a plaintext header (or vice-versa) page 1 becomes garbage -> 'file is not a database' (the incident's SQLITE_NOTADB). The corruption is silent: import reports success.
- **Reachable:** Reachable on current code. migration.rs apply_import movable_files swap (557-578) provably moves only the `.sqlite` file; open_archive_for_migration (631-640) is a thin alias for open_archive_connection (schema.rs:98) which has no sidecar scrub/validation; run_migrations (schema.rs:323-348) sets journal_mode=WAL and lets SQLite auto-replay. No concurrency needed — only a leftover hot -wal from a prior crash, which is precisely the failure class the project already hit once.
- **Existing test coverage:** none — migration/fault_tests.rs only asserts absence of `.bak-*` sidecars (assert_no_backup_sidecars, lines 63-72); no test injects a leftover `history-vault.sqlite-wal` before apply_import and checks the imported DB's integrity.
- **Proposed test:** In migration/fault_tests.rs: build a real bundle from archive B containing a unique row NEW. Separately build archive A (same page size + same at-rest mode) holding row OLD, write+commit a transaction with `PRAGMA wal_autocheckpoint=0` so OLD lives only in `history-vault.sqlite-wal`, then copy A's main DB + its `-wal`/`-shm` over the live target paths (simulating a post-crash hot WAL). Call apply_import(bundle) with confirm_overwrite. Open the installed archive and assert (a) `PRAGMA integrity_check` == 'ok', (b) the NEW row is present, (c) the OLD row is absent. Fails on current code because A's foreign -wal is replayed into B.
- **Proposed hardening:** Treat each SQLite DB as the unit (db + -wal + -shm + -journal): before installing a staged DB, move/scrub the live target's sidecars too (mirror at_rest.rs remove_stale_sidecars, archive/at_rest.rs:271-275, and include them in the `.bak` set so recovery stays complete); and BEFORE open_archive_for_migration, assert no pre-existing `<target>-wal` exists (or delete it) so SQLite cannot replay a foreign WAL into the imported DB.

The claim's core (only .sqlite moved -> stale -wal -> silent replay into the new DB; SQLITE_NOTADB for mode-mismatch) is code-accurate and mechanically proven. ## Per-dimension summaries

### incident-rootcause

Root cause of the incident {history-vault + source-evidence encrypted on disk, config.archive_mode="Plaintext", history-vault first-16-byte salt zeroed, every open = SQLITE_NOTADB}: the rekey flow in vault-core/src/archive/maintenance.rs::rekey_archive converts the canonical files on disk BEFORE persisting config, and does so with ZERO durability barriers.
