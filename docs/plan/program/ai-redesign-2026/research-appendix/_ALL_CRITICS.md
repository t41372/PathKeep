# All adversarial critics

## [llm-integration] LLM integration (post-litellm 2026)

**FINAL REC:** Adopt a TWO-LAYER design, with PathKeep owning a thin internal trait so no third-party type is load-bearing.

LAYER 1 — Transport (own it): define PathKeep's own `LlmProvider` trait (chat, stream, tool-call, structured-output, capability flags). Back it initially with genai (v0.6.x, pinned + vendored + risk-assessment, single-maintainer caveat) because it is the best feature-per-line pure-Rust, zero-native-dep, multi-provider fit and keeps model IDs as plain user-config strings. Keep async-openai available as the OpenAI/Responses adapter if/when Responses-specific features (built-in tool loop, better cache) are wanted. HARD RULE: OpenAI-compatible Chat Completions is the FLOOR only for OpenAI-shaped and LOCAL servers (Ollama, LM Studio, llama-server); Anthropic and Gemini MUST go through NATIVE adapters (genai already provides these) so structured output (strict) and prompt caching actually work — do NOT route Claude through its OpenAI-compat shim. Treat Responses as an optional richer adapter, never the baseline. Wrap API keys in secrecy::SecretString; never serialize them into traces/sidecars.

LAYER 2 — Agent/tools: model PathKeep's history/evidence tools as an IN-PROCESS MCP server using the official rmcp crate (v1.7.0, Anthropic-maintained → trust exemption). This keeps the evidence-citing tool surface model-agnostic and reusable across any provider. External MCP servers = hard default-off, per-server consent (egress boundary). Use constrained/structured output via each provider's NATIVE schema mode, but ALWAYS keep a serde validate-and-repair fallback (constrained = parseable, not semantically correct).

LOCAL INFERENCE policy (the envelope-binding decision): EMBEDDINGS and CHAT are asymmetric. Embeddings (Qwen3-Embedding-0.6B Q8 ≈ 400MB, 20-100ms/chunk CPU) are cheap enough to bundle in-app. CHAT is NOT — default to external providers (Ollama/LM Studio the user already runs; zero supply-chain burden, full sovereignty). If in-app chat ships at all, do it as a llama-server Tauri SIDECAR (not in-process mistral.rs) for OOM isolation and reclaimable RAM, bound to 127.0.0.1, behind: explicit consent + a hard pre-spawn RAM check + a mutex that FORBIDS inference during bulk import. Cap to 0.6B-4B Q4 models. Everything AI is OFF by default; PathKeep stays fully usable with no LLM and no embeddings, per the non-AI deterministic intelligence already in place. Note: the Rust LLM-client landscape is fragmenting, not consolidating (genai 806★ / rig 7.7k★ / mistral.rs 7.3k★ / litellm-rust ~4★) — owning the trait boundary is what insulates PathKeep from that churn.
**scale_failure:** The LLM-transport layer itself does NOT break at 14.4M rows — it is pure HTTP and stateless; genai/async-openai add no native deps and negligible RAM. What breaks is OPTIONAL IN-APP INFERENCE colliding with the data workload on 4-core/8GB-no-GPU. Concrete memory math (sources below):

(1) Steady-state floor before any LLM: OS + Tauri/WebView2 (Chromium) on an 8GB box realistically eats 1.5-2.5GB; SQLCipher page cache + the import/search working set wants every spare GB. That leaves ~4-5GB usable headroom on a fresh boot, far less mid-import.

(2) An in-app CHAT model is the real risk. Per 2026 local-LLM guidance, a 4B model at Q4_K_M is ~2.5GB weights, and a 7B Q4_K_M is ~4-5GB weights + 1-2GB KV cache + overhead (corsair/localllm/aimagicx 2026). On 8GB, a 7B leaves essentially nothing for the WebView + a live 14.4M-row import — that path swap-thrashes (the n1n.ai 2026 Tauri+llama.cpp post explicitly warns "loading a 16GB model on an 8GB machine -> swap-thrashing"). Even a 4B Q4 (~2.5GB + ~1GB KV) plus a 2GB WebView plus an active import working set is right at the cliff. CPU-only throughput for these is only ~15-20 tok/s (aimagicx/sitepoint 2026) — usable for short evidence-summaries, painful for long agent loops.

(3) THE CRITICAL CONCURRENCY FAILURE the research underweights: 4 physical cores. llama.cpp/mistral.rs CPU inference is memory-bandwidth- and core-bound and will happily saturate all 4 cores. A 14.4M-row bulk import is ALSO CPU/IO-bound (SQLCipher AES + FTS5 tokenization + index build). Run both at once and you get neither — the import stalls and the "off-thread, never freeze the main thread" guarantee is preserved for the UI thread but the machine itself is pinned. mistral.rs IN-PROCESS makes this worse: an inference OOM kills the canonical-DB process. The research's own caveat (CPU perf "under-documented vs CUDA benchmarks") is confirmed — I could find NO documented CPU-only tok/s for mistral.rs; that is a real adoption risk, not a footnote.

(4) Conclusion for scale: the transport abstraction scales fine; in-app inference must be (a) OFF by default, (b) a SIDECAR process (llama-server) not in-process, so OOM is isolated and memory is reclaimable, (c) mutually exclusive with bulk import (gate it: no inference while an import job is running), and (d) capped at small models (Qwen3 0.6B-4B class). The honest default is to NOT bundle an LLM at all and steer users to an external Ollama/LM Studio they already run.

## [embedding-inapp] In-app embedding inference (Rust, CPU)

**FINAL REC:** Decouple the decision into ENGINE, MODEL TIERS, and INDEX — the research conflated them, and the binding constraints (multi-day embed wall-clock; index RAM > machine RAM) are NOT solved by engine choice.

ENGINE: Adopt candle (via fastembed's `qwen3` candle feature) as the no-native-binary default engine — it is the only path that cleanly passes the supply-chain rules (pure-Rust kernels, 20.5k stars, Apache-2.0/MIT, no downloaded C++). Explicitly REJECT ort for the default path: its download strategy is banned, its compile strategy is the banned build.rs-compiles-C++ pattern, and its system strategy forces you to vendor an ONNX Runtime build (ADR-level commitment). Keep ort as a documented power-user/optimization path only if profiling proves candle is too slow AND someone owns the ONNX Runtime static build. Pin fastembed to an exact version and add a re-eval cadence (it ships ~weekly). Add SHA256 hash-pinning + offline model bundling yourself (fastembed has none) and support a fully-offline first run (ship or sideload the default model, never require a network fetch).

MODEL — two tiers, model-agnostic in UI: (1) DEFAULT FAST TIER for the 14.4M one-shot import and the 'just works on a weak box' experience: a STATIC embedding (model2vec-rs + potion-multilingual-128M, 100% Rust, ~8000 docs/sec single-thread, 8-30 MB). This turns a multi-day embed into ~tens of minutes and needs zero native deps. (2) QUALITY TIER (opt-in, background, resumable): Qwen3-Embedding-0.6B (Apache-2.0 default, as mandated), int8/quantized, run as a throttled off-thread upgrade pass that re-embeds in the background and swaps the index in atomically. Expose model choice but never hardcode IDs in copy. Re-baseline ALL throughput claims on the actual 4-core/3GHz target before promising any import ETA.

INDEX (the part the research missed and the real blocker): vectors live ONLY in a rebuildable sidecar, never canonical SQLite. Use Matryoshka truncation applied to the fp32 vector (default 256 or 512 dims) THEN BINARY quantization (1 bit/dim) for the resident index: 14.4M x 512-bit = ~0.9 GB, x 256-bit = ~0.45 GB — fits 8GB with room for the WebView, SQLCipher cache, and the running embedder. Search = Hamming top-K over binary codes (cheap), then fp32/int8 RESCORING of the top candidates (~96% recall recovered). Build it as a DISK-RESIDENT/mmap ANN (DiskANN/Vamana, pure-Rust) so RAM stays bounded regardless of corpus growth; do NOT use in-memory HNSW (55-61 GB fp32 / 14.7 GB uint8 both exceed the machine). Do not use brute-force sqlite-vec scan at this scale (O(n)/query).

OPTIONALITY GATE (non-negotiable per AGENTS.md): ship SQLite FTS5 + bm25 + trigram as the always-on, zero-AI default search. It must fully serve search over 14.4M rows while the semantic index builds in the background (potentially for days), and the product must remain complete with AI disabled. Semantic search is purely additive, off-thread, chunked, resumable, and never blocks the main thread or the import. Add a quality gate that detects a degenerate index (e.g. all-zero or near-random vectors, guarding against the known GGUF Qwen3 bugs) before promoting a freshly built index to live. Every AI-surfaced result must cite the real history rows it came from (transparency over black-box).
**scale_failure:** The research's biggest blind spot is treating the EMBEDDING ENGINE as the hard problem when the binding constraint at 14.4M docs on 4-core/8GB is (a) total embed wall-clock and (b) the vector INDEX memory, neither of which any single engine fixes.

EMBED WALL-CLOCK (one-shot 14.4M import): The only cited Qwen3-0.6B CPU latency, ~30ms/doc, is on an Intel i9-13900HK (6 P-cores + 10 E-cores, 45W) per the HF discussion — NOT the 4-core/3GHz/8GB target. Qwen3-0.6B is a 28-layer DECODER with last-token pooling, ~600M params; on 4 weak cores expect ~50-120ms/short doc even at int8. Math: 14.4M docs x 60ms / 4 cores (ideal linear scaling, which rarely holds with shared memory bandwidth) ~= 60 hours of pure compute for a single full pass; realistically a multi-day background job. candle CPU kernels are generally slower than tuned ONNX Runtime for the same model (corroborated qualitatively, not with a clean 4-core docs/sec number — flagging that uncertainty), so the recommended candle/qwen3 path is the SLOWEST viable embedder for the heaviest workload. This is survivable only as a throttled, resumable, off-thread background job that may run for days — the design MUST treat first-index as eventually-consistent, never blocking, with the deterministic FTS5/bm25 path serving search in the meantime.

INDEX MEMORY (the real wall): The research understates this badly. 14.4M x 1024-dim fp32 ~= 55-61 GB RAM for an in-memory HNSW (3.8 GiB/M for vectors + 0.15-0.6 GiB/M graph overhead, M=16-32) — ~7x the entire 8GB budget. The research's own 'int8/uint8 x1024 ~= 14.7 GB' figure is ALSO larger than 8GB RAM, so a flat in-RAM uint8 index does not fit either. And the cited 'MRL-256 uint8 ~= 3.7 GB' is partly built on a false premise: the specific electroglyph uint8 ONNX model the research leans on explicitly has NO MRL/dimension-truncation support (its card says so) — you cannot truncate that model's output to 256-dim. MRL truncation must happen on the fp32/bf16 hidden state BEFORE scalar quantization (e.g. via the candle/safetensors path or a model exported with MRL), so 'uint8 + MRL-256' is not a free combination on an off-the-shelf uint8 ONNX export. Even at MRL-256 uint8 = ~3.7 GB resident, plus the ~0.6 GB model, plus SQLCipher page cache, plus React/Tauri/WebView (Chromium WebView alone is commonly 300-700 MB), plus HNSW graph links, you are over budget during the import that is itself running the embedder.

WHAT ACTUALLY BREAKS, concretely: (1) Building a standard in-memory HNSW OOM-kills the process partway through import. (2) Holding fp32 vectors to feed HNSW construction OOMs even faster. (3) Naively truncating an off-the-shelf uint8 export to 256-dim silently corrupts recall. (4) The import job, the embedder model, the WebView, and SQLCipher cache contend for the same 8GB, so an embed run that 'fits' in isolation tips into swap under real app load, and on a 4-core box swap-thrash also starves the UI thread the project forbids from freezing. (5) llama.cpp GGUF path: issue #14234 (~20% retrieval degradation, still bug-unconfirmed/unresolved at fetch time) and #20085 (all-zeros with embedding+reranking) mean the GGUF route can silently ship a broken index at this scale.

The fix the research missed: the only configuration that fits 8GB is QUANTIZED + DISK/MMAP index. Binary quantization gives 32x compression: 14.4M x 1024-bit = ~1.8 GB (or MRL-512 binary ~= 0.9 GB), and binary preserves ~96% retrieval with an fp32 rescoring pass (Hamming distance is ~2 CPU cycles). Pair that with a memory-mapped/disk-resident ANN (DiskANN-style: PQ/binary codes cached in RAM, full graph + residuals on SSD) so RAM stays bounded regardless of corpus size. That combination — not the choice between candle and ort — is what makes 14.4M actually run on this box.

## [embedding-external] External embedding providers

**FINAL REC:** Build ONE OpenAI-compatible HTTP adapter as the universal external path (covers Ollama native + /v1, LM Studio, TEI, and most cloud in a single thin in-house reqwest+serde client — do NOT add async-openai), plus tiny dedicated adapters only for non-OpenAI-shaped clouds (Gemini task_type/single-input, Voyage/Cohere input_type+output_dtype). Behind that, an EmbeddingProvider trait whose config is a versioned FINGERPRINT = {provider, model_id, effective_dimension, output_dtype, normalized, instruction_template, version} that is PROBED at runtime (never trusted from config — read the actual returned vector length, assert it every batch) and stamped onto the sidecar index; any mismatch = stale = explicit, warned, resumable re-embed with dual-index query during migration. Hard invariants in the trait: carry a query-vs-document role through every call, and L2-renormalize after any MRL truncation.

For the in-app DEFAULT (the part the research under-specified): ship the uint8 Qwen3-Embedding-0.6B ONNX (624 MiB, ~1% loss, 25% faster CPU, native uint8 output) at MRL-512, written into a DISK-RESIDENT memory-mapped quantized sidecar index (USearch view-mode is the leading candidate — vendored C++ build.rs, Apache-2.0, i8/b1 quant, 'serve 100GB on 8GB RAM'; gate it behind a written risk assessment since it's 4.2k stars, with a pure-Rust mmap fallback as plan B). This is the ONLY combination that fits 14.4M in 8 GB: 512-dim uint8 ≈ 0.5 KB/vector → ~7.4 GB of codes that the OS pages from SSD, vs the impossible 54-59 GB of in-memory f32@1024 the research implicitly assumed. NEVER use in-memory full-precision HNSW at this scale, and default to int8/uint8 NOT binary (binary materially hurts Qwen3 and only recovers with full-vector rescoring).

Treat first-embed as a first-class operational subsystem, not a config flag: the one-shot 14.4M embed is a 12-100h CPU-bound job (extrapolated from ~30 ms/doc on a 20-core i9; worse on 4 cores), so it must be chunked, checkpointed/resumable, cancellable, off-main-thread, thermally/battery throttled, with visible progress+ETA — and the app must stay fully usable on deterministic FTS5+trigram+bm25 the entire time AI is off, absent, or mid-build (AI strictly additive, consent-gated, off by default). Cloud embedding providers are incremental/query-time only, NEVER for bulk-archiving the full history (throughput + sovereignty); Gemini's single-input-per-request makes it unfit for bulk by construction. Guard quality with golden-vector + recall@k canaries in `bun run check` so silent instruction/normalization regressions can't ship.
**scale_failure:** The leading recommendation has TWO independent scale failures at 14.4M docs on 4-core/8GB, and the research underweighted both by treating the index as an afterthought ("hnsw_rs + quant") rather than the binding constraint.

FAILURE 1 — INDEX MEMORY (the dominant one). The research's headline plan is in-app Qwen3-0.6B at 1024-dim into an HNSW sidecar. Raw float32 vectors alone = 14.4M × 1024 × 4B = ~59 GB (Zilliz/Weaviate math: ~3.8 GiB per 1M vectors → ~54-59 GB), plus HNSW graph (M=16-32) adds 2-8 GB. That is ~7x the ENTIRE 8 GB RAM budget, before the OS, SQLCipher page cache, React/WebView (Tauri WebView2/WKWebView alone is 300-800 MB), and the embedding model itself (624 MiB-1.2 GB resident). An in-memory HNSW of full vectors is physically impossible here. The research mentions quantization as a "lever" but never confronts that it is MANDATORY, not optional, and never did the memory math. Even MRL→512 + int8 = 14.4M × 512 × 1B ≈ 7.4 GB of codes — still does not leave room for the graph + app + model in 8 GB if held fully in RAM. The ONLY designs that fit are: (a) memory-mapped / disk-resident index where the OS pages quantized vectors on demand (USearch `view=True`, or DiskANN/RaBitQ-style where only compressed codes/graph are hot), or (b) int8/uint8 at 512-dim with an mmap'd graph so resident set is a few GB, not the whole index. Source: Zilliz memory-overhead FAQ (faq, current 2026), Weaviate HNSW+PQ blog, USearch docs ("memory-map ... serve a 100GB index on a machine with 8GB RAM", converter.brightcoding.dev 2026 + unum-cloud.github.io v2.25.3 2026-05).

FAILURE 2 — ONE-SHOT EMBEDDING THROUGHPUT (the wall-clock one). The research cited 5-14k sentences/sec, but that is MiniLM-L6 (22M params, static-ish). Qwen3-0.6B is ~27x larger. Real reported CPU figure: ~30 ms PER embedding for a MINIMAL request on a 20-logical-core mobile i9-13900HK with IPEX + bf16 + core pinning, and that user had to cap input to 1/4 max-tokens to avoid OOM on warmup (HF discussion #48, 2026). On a 4-core 3GHz box with no IPEX tuning and longer real inputs, expect materially worse than 30 ms — call it 30-100 ms/doc realistically. At even an optimistic 25 ms/doc single-stream, 14.4M docs = 360,000 s ≈ 100 HOURS. With 4 cores batched you might reach 4-8x → still ~12-25 hours of continuous CPU saturation for the one-shot import, during which the machine is hot, fans max, and battery-hostile. The uint8 ONNX build (electroglyph/Qwen3-Embedding-0.6B-onnx-uint8, HF 2026) gives ~25% CPU speedup (34.6s vs 46.1s/chunk) — helps but does not change the order of magnitude. CONCLUSION: in-app embedding of the full 14.4M archive is a multi-hour-to-multi-day background job, NOT something to run synchronously or block import on. The research's framing of in-app default as the happy path hides this. It must be designed as a resumable, checkpointed, interruptible, throttled background job with a visible progress/ETA, and the app must be 100% functional (deterministic FTS5/trigram search) the entire time embeddings are absent or partial.

SECONDARY at scale: (a) re-embed on model switch means re-running that 12-100h job over all 14.4M rows — the "fingerprint mismatch → rebuild" pattern the research likes is correct but the cost is brutal here, so partial/lazy/background re-embed with dual-index query during migration is not a nice-to-have, it is required. (b) Cloud providers for the FULL archive are a non-starter on throughput AND sovereignty: Gemini is single-input-per-request (14.4M sequential calls + rate limits = days and a privacy breach of the whole history). Cloud is only sane for incremental/query-time embedding, never bulk archive.

## [vector-store-memory] Memory-constrained ANN at 14.4M on 8GB

**FINAL REC:** Build a purpose-built, pure-Rust IVF + RaBitQ + disk-rerank sidecar — NOT usearch-HNSW, and NOT a disk-graph (DiskANN/Vamana) — sized in tiers, with embedding generation treated as the primary cost.

CONCRETE STACK:

1. DEFAULT = no ANN at all below a threshold (~1M vectors): int8/uint8 FLAT SIMD scan over RAM-resident codes (1M x 1024 B = ~1 GB). Exact recall, zero build, zero corruption risk. This covers most users for years and honors 'AI is optional/additive.'
2. ABOVE the threshold: IVF (inverted-file) routing. Builds in minutes on 4 cores (vs HNSW hours-days); a query reads a few WHOLE posting lists = sequential-ish, batchable SSD reads instead of dozens of dependent QD1 random hops; trivial incremental append + minutes-scale periodic re-cluster. Keep the centroid layer tiny in RAM (tens of MB, per arXiv:2602.21514 MemGraph 0.1%-vertices / 30-50 MB pattern).
3. QUANTIZATION = RaBitQ (1-bit + 2 corrective scalars), ported to portable Rust SIMD. Default to Matryoshka-512 codes (~0.9-1 GB RAM for 14.4M) for the 8 GB box; expose 1024-dim as a 'high-recall' opt-in. RaBitQ over PQ because: no k-means codebook to train (cheap incremental insert), theoretical O(1/sqrt(D)) error bound, fast estimated distances.
4. RERANK tier = Qwen3-0.6B uint8 ONNX 1024-dim vectors on disk (14.75 GB, ~1% NDCG loss) — NOT f32 (59 GB). Oversample 2-4x candidates from IVF/RaBitQ, mmap-read their uint8 vectors, rescore, return top-K. 4x less rerank I/O than f32, materially friendlier to the consumer SSD and tiny page cache.
5. HOT/COLD split for the 60-year horizon: keep recent (~1-3M) vectors' codes hot in RAM, long tail cold on disk; most history queries are recent.

NATIVE-DEP / SUPPLY-CHAIN: this stack is pure Rust (RaBitQ math + IVF + mmap via memmap2) with NO new C/C++ native dependency beyond the already-required SQLCipher/OpenSSL/ONNX-Runtime toolchain, which is the cleanest possible supply-chain outcome and beats usearch (4.2k stars, needs a risk note) and any <=10-star DiskANN port. ONNX Runtime (via the `ort` crate, vendored) covers Qwen3-0.6B INT8/uint8 inference. If a hand-rolled IVF is judged too much work for v1, hnsw_rs (pure Rust, ~242 stars, risk-note) is the plan-B router — but ONLY with RaBitQ codes + uint8 disk rerank layered on, and accepting its hours-scale CPU build.

NON-NEGOTIABLES FOR THE ENVELOPE: (a) embedding the archive is a resumable, throttled, OFF-thread background job, never a blocking precondition — the app stays fully usable on FTS5 + deterministic intelligence while 0% embedded; (b) all rerank/disk reads are off the UI thread, chunked, cancellable, with skeletons; (c) every AI answer cites the real history rows it reranked (transparency); (d) measure on actual 4-core/8GB/consumer-SSD hardware before claiming 'fast' — server SSD/Xeon benchmarks do not transfer. Reconsider a disk-graph index ONLY if a >6k-star, reputable-maintainer Rust DiskANN with built-in quantization and cheap deletes appears AND it beats IVF on cold-cache QD1 latency on the target box.
**scale_failure:** Concrete breakage on a 4-core/3GHz / 8GB / no-GPU / consumer-SSD desktop at 14.4M x 1024-dim Qwen3-0.6B vectors:

(1) THE EMBEDDING WALL IS THE REAL FIRST FAILURE, AND THE RESEARCH BARELY QUANTIFIES IT. Before any index exists you must run 14.4M documents through a 0.6B-param model. The one real CPU data point found is ~30ms/embedding for a _minimal_ request on an Intel i9-13900HK (8 P-cores + 8 E-cores, far stronger than the 4-core target) per the Qwen HF discussion (huggingface.co/Qwen/Qwen3-Embedding-0.6B/discussions/48). The 44K tok/s figure (github.com/jakedahn/qwen3-embeddings-mlx) is Apple-Silicon MLX (GPU/ANE), NOT a 4-core CPU, and must not be used for envelope math. Realistic 4-core CPU throughput for a 0.6B encoder with real history-page text is on the order of tens of docs/sec even with INT8 ONNX (2-4x over f32, per electroglyph/janni-t model cards). At, say, 50 docs/sec, 14.4M docs = ~80 hours of continuous CPU; at 20 docs/sec it is ~200 hours. This is the dominant cost, dwarfs index build, and means full-archive embedding can NEVER be a blocking foreground operation — it must be an opt-in, resumable, checkpointed, throttled background job, and the product must be fully useful (deterministic + FTS5) while only a fraction is embedded.

(2) RAW SIZE: 14.4M x 1024 x f32 = 58.98 GB; int8 = 14.75 GB (still > 8GB, the research is right that even int8 RAM-resident is impossible). 1-bit RaBitQ codes = 14.4M x 1024 / 8 = 1.84 GB codes BUT RaBitQ/LanceDB store TWO corrective scalars per vector; LanceDB measures ~136 bytes/vector (lancedb.com/blog/feature-rabitq-quantization), i.e. ~1.96 GB, not the bare 1.84 GB. Add an HNSW routing graph (M=16 => ~1.5-2 GB resident, NOT mmap-able cheaply) and you are at ~3.5-4 GB just for routing structures, leaving little of the 8 GB for OS + Tauri/WebView (Chromium easily 0.5-1.5 GB) + SQLCipher page cache + the embedding model (~0.6 GB) + rerank read buffers. (3) THE I/O WALL IS THE REAL SERVING FAILURE FOR GRAPH INDEXES, AND IT IS UNDERWEIGHTED. arXiv:2602.21514 (Feb 2026) shows disk-graph ANN is 70-90% I/O-bound and SATURATES SSD random-read IOPS at 100M scale on a 4TB NVMe doing 819K 4K-random-read IOPS. The catch: that 819K IOPS requires deep queue depth. A desktop ANN rerank issues dependent reads largely at QD1, where consumer NVMe latency is ~10-70us/read (digitalcitizen/simplyblock 2025-2026), i.e. only ~14K-100K _effective_ IOPS per thread, 10-50x below the benchmark drive. A DiskANN/Vamana or usearch-HNSW-view beam search does dependent random hops (each hop blocks on the previous), so on a 4-core desktop with a mid-range SSD, p95 query latency is realistically tens of ms, and concurrent queries contend on the same few cores + one SSD queue. This is why the research's "DiskANN is architecturally ideal" framing is misleading for THIS hardware.

(4) BUILD TIME: HNSW is hours-to-days on CPU at scale (faiss wiki / GSI: 100M in ~1.5h on a 56-core Xeon Platinum 8480; eBay 160M in 3-6h). Scaled to 4 cores that is roughly 14x worse => an HNSW build over 14.4M could be many hours to >1 day. IVF builds in seconds-to-minutes (ANN-Benchmarks: FAISS-IVF ~2s vs HNSW ~5h on the same data). For a rebuildable sidecar that must survive corruption/version bumps and grow with the archive, a multi-hour-to-day HNSW rebuild is a product-level failure; an IVF rebuild is not.

(5) usearch b1 IS NOT A COSINE 1-BIT STORE. usearch's b1x8 single-bit path is valid ONLY for binary metrics (Hamming/Jaccard) per its docs (unum-cloud.github.io/USearch). You cannot drop Qwen cosine vectors into usearch as b1 and get cosine search; you must binarize yourself and rerank. So usearch buys you an HNSW-over-binary index (random-read traversal, hours to build) but NOT the RaBitQ math. The research conflated "usearch supports b1" with "usearch gives you RaBitQ+rerank," which is false.

## [agent-architecture] Long-running agent architecture (2026)

**FINAL REC:** Roll your own thin Rust agent loop — the research is right on this — but reframe the priorities. The loop is the easy 10%; the 90% is (1) the vector-search TOOL the agent calls, which the research treats as solved and is not, and (2) replay-safe durability and context discipline.

Architecture for PathKeep:

- LOOP: hand-written tokio while-loop in the Rust backend. Use a NARROW provider abstraction — prefer the small single-purpose genai crate (with a written risk assessment, 806 stars) OR hand-roll 3-4 provider JSON shapes; treat Rig as optional/at-threshold, not a dependency to lean on. Stream tokens to the UI, support hard cancellation, and enforce a per-run step + token/cost budget. Detect tool-calling capability and degrade to deterministic search when the user's model can't tool-call.
- TOOLS as plain Rust traits (search, fetch-evidence, aggregate, write-note, update-todo, bounded sub-search), 6-10 of them, static cached schema. Adopt rmcp ONLY at the EXTERNAL-MCP boundary, behind consent — not for calling your own in-process functions. Be honest that rmcp is 0.x / ~2.7k stars and clears the gate on 'reputable org', not stars.
- CONTEXT: implement selective recency-pruning (drop whole stale tool-result pairs) + lightweight summarization as the PRIMARY strategy (per arXiv 2606.10209), reserving expensive LLM auto-compact as a last resort. Keep working tokens well under ~32k (Context Rot). Carry row IDs + counts, fetch evidence just-in-time, NEVER inline large result sets. Persist a citation table (row ID -> evidence) in the sidecar so citations survive compaction — directly serving the transparency requirement.
- DURABILITY: extend the existing lease-based job queue over a sidecar SQLite (rebuildable derived state, NOT the canonical encrypted DB). Do NOT adopt Temporal/Restate/DBOS/the iopsystems 'durable' crate (all server/Postgres/WASM). Contract from day one: journal each model + tool output BEFORE observing it; on resume replay journaled bytes, never re-invoke the model (avoids re-charging the user and divergence); idempotency keys per tool call; a long-pause state for PME approvals.
- SUB-AGENTS: support but cap concurrency at ~2, run at lower priority, feed them PRE-FILTERED deterministic results, and forbid them from triggering bulk embedding — otherwise they break the UI-fluidity hard requirement on 4 cores.
- MEMORY: implement the lightweight Anthropic memory-tool PATTERN over the sidecar, reusing the existing embedding stack for recall, consent-gated and user-deletable.

Hard dependency this area inherits: the agent's semantic-search tool CANNOT be in-RAM HNSW (14.4M x 1024-dim fp32 HNSW ~61GB; even int8 512-dim ~7.4GB at the 8GB ceiling). It must be disk-resident ANN with MRL-512 + int8 quantization (e.g. usearch with its vendored-C-backend crate, which PathKeep's rules explicitly ALLOW, since the archived pure-Rust DiskANN crate fails the trust gate). The agent must stay fully useful in deterministic FTS5/bm25/trigram mode while the multi-hour-to-multi-day (≈10-40h at ~100-400 docs/s) one-shot embedding backfill runs — and that backfill IS the headline use case for the durable job queue. Bottom line: ship the simple loop, spend the engineering budget on replay-safe durability, recency-pruned context with a persistent citation table, and a disk-quantized vector tool — and explicitly cap sub-agent concurrency to protect the main thread.
**scale_failure:** The agent loop itself never breaks at 14.4M rows — it is a thin while-loop making remote HTTP calls. What breaks is the TOOLING the agent calls, and the research underweights this by treating the vector layer as a solved sibling concern. Concrete failure modes on 4-core/8GB/no-GPU:

(1) VECTOR INDEX RAM BLOWUP — the real wall. Qwen3-Embedding-0.6B is 1024-dim. In-memory HNSW at fp32: 14.4M x (1024x4 + ~40 neighbors x 4) = 14.4M x ~4256B = ~61GB. Even at 768-dim it is ~46GB; the search result's 768-dim/40-neighbor figure of ~3232B/vector x 14.4M = ~45GB. Either way, 6-8x over the 8GB budget. Raw fp32 vectors alone (14.4M x 1024 x 4) = ~59GB on disk. So pure-in-RAM HNSW is categorically impossible and the agent CANNOT assume a fast in-memory semantic-search tool exists. The viable path is disk-resident ANN (DiskANN/Vamana/SPANN) with int8/scalar-quantized vectors held in RAM as a guide: int8 at 1024-dim = ~14.7GB (still too big to fully reside) so you also need MRL truncation to 512-dim (Qwen3 reports only -1.4% retrieval loss) -> int8 512-dim = ~7.4GB, at the ceiling and competing with the app, OS, and the LLM HTTP buffers. Realistic design: 512-dim + int8 + DiskANN with the graph on SSD and a PQ/quantized in-RAM guide (~1-2GB), reranking the top candidates against fuller-precision vectors read from disk. This means every agent semantic_search tool call is a disk-bound operation with p95 in the tens-to-hundreds of ms, NOT the 10ms in-RAM figure the research implies.

(2) ONE-SHOT BULK EMBED is the worst operation, and the agent design must never block on it. At Qwen3-0.6B CPU latency of ~380ms/query single-thread (GPU T4 = 85ms), even with int8 ONNX 2-4x speedup and 4-core batching, sustained throughput is realistically ~100-400 docs/sec. 14.4M docs => ~10-40 hours of CPU-saturated embedding for the initial import. This must be a resumable, checkpointed background job (exactly the durable-execution case), chunked so the UI never freezes, and the agent must remain useful in DETERMINISTIC (FTS5/bm25/trigram) mode while the embedding backfill is still running for months of history. The research's agent section never states that the agent's semantic tools are partially-available during a multi-hour/day backfill — a correctness gap for any 'search my whole history' request.

(3) PER-TURN LATENCY STACKS. Each agent turn = remote model RTT (seconds) + tool exec. If a tool triggers an embedding of the query (~380ms CPU) + disk ANN (tens of ms) + evidence row fetch from SQLCipher (decryption cost), a 10-tool-call investigation is dominated by remote model latency but the LOCAL contention (embedding the agent's sub-queries, SQLCipher decryption, ANN disk reads) competes with the UI thread's data needs on only 4 cores. Sub-agent fan-out (the research's 'canonical scaling lever') is actively DANGEROUS here: 3 concurrent sub-agents each embedding queries + hitting the disk ANN can saturate all 4 cores and stall the very UI the project treats as a hard requirement. Concurrency must be capped at ~2 and sub-agents must consume pre-filtered deterministic results, not run raw scans.

(4) CONTEXT, NOT RAM, is where the agent loop fails. Chroma's Context Rot research and 2026 LoCoMo show 11/12 models drop below 50% of short-context accuracy past ~32k tokens. A 'find everything about X across 14.4M rows' query can return thousands of candidate rows; dumping even top-200 rows (URL+title+timestamp+snippet) blows past the useful window long before any hard limit. The agent MUST carry row IDs + counts and fetch evidence just-in-time, never inline large result sets.

## [agent-tools-skills-codeexec] Agent tools, skills & code execution

**FINAL REC:** Build the agent surface as THREE LAYERS, default-safe and progressively unlocked, with the vector-index decision (not the sandbox) treated as the load-bearing choice.

LAYER 0 - Retrieval floor (ship first, always on, AI-optional-safe): A small set of ~8-12 BOUNDED, read-only structured tools over the canonical DB and sidecars. Every tool result carries coverage/freshness metadata so the agent degrades gracefully when the index is partial/absent.

LAYER 1 - Semantic/hybrid (additive, consent-gated): Default index is Qwen3-Embedding-0.6B MRL-truncated to 256-d, BINARY-quantized in RAM (~0.5-1.9GB depending on dim), with a float32/int8 rerank pass over top-K from an mmap'd sidecar - this is the only scheme that fits 14.4M vectors in the 8GB envelope while holding >95% recall. Add a small reranker (Qwen3-Reranker-class) for recall recovery. Plain in-RAM float32 HNSW is OFF THE TABLE at this scale (55-61GB) and must be stated as such. Keep DiskANN/Vamana-on-SSD as a documented fallback IF and only if a high-trust vendored Rust crate exists (else ADR). Embedding/index build runs chunked on the off-main-thread ai worker, resumable, never blocking the UI, and the design assumes it may take hours and be incomplete.

LAYER 2 - Code-mode (OPT-IN, capability-gated upgrade, NOT the default): When the user configures a capable model (local 8B+/Devstral-class or a cloud provider), let the agent write JS that fans out many BM25/vector/hybrid/granularity queries and joins/filters LOCALLY, returning only distilled top-K rows with visit_ids. Capability-based: ZERO ambient authority; the ONLY host functions are the same bounded read-only query API from Layer 0 - no DB handle, no SQL string, no fs/net. Do NOT ship MicroPython-in-WASM (its own author marks it alpha/risky); offer JS only, or no code-mode on the Python path. **scale_failure:** The research's biggest blind spot is that it treats the SANDBOX engine choice as the hard problem while the actual scale wall is the VECTOR INDEX the agent's tools query. Concrete failures at 14.4M docs on 4-core/8GB:

1. HNSW float32 is physically impossible in 8GB. ## [hybrid-search-rerank] Hybrid search + reranking

**FINAL REC:** Ship a 3-stage hybrid pipeline, but build it around quantization and a disk-resident index — the parts the original research omitted are the parts that decide whether it works at 14.4M on 8GB.

ARCHITECTURE (CPU, no GPU):

1. LEXICAL leg: keep FTS5 + bm25() as an EXTERNAL-CONTENT table (no content duplication), with hard candidate caps and exact-host/URL short-circuit. This is the always-available default and works with zero AI. Mitigate the `ORDER BY rank LIMIT` blowup (20s at 6M rows) with bounded candidate sets and query gating.
2. DENSE leg: Qwen3-Embedding-0.6B (default, model-agnostic in UI) -> 1024-d, optionally MRL-truncated to 512. Store BINARY-quantized vectors (128 bytes/vec -> ~1.8GB for 14.4M, fits RAM) for first-stage Hamming retrieval, plus int8 vectors on SSD for rescore. Use USEARCH (mmap 'view' index + b1x8/i8 quantization, vendored C++ via cxx-build, Apache-2.0) — write the required risk assessment for its 4.2k-star count. Pipeline: binary ANN -> int8/float32 rescore of top ~200 (this rescore is mandatory for recall AND is most of the quality the research attributed to a separate reranker). This keeps RAM ~2-3GB, not 53GB.
3. FUSION: RRF (k=60) as the principled default — PathKeep has no relevance labels to tune convex weights, so RRF is the right choice, not a compromise. Expose an optional lexical/semantic weight as a power-user knob only.
4. RERANK: OPTIONAL, off by default, on the final fused shortlist (<=50), debounced, never per-keystroke. Default multilingual reranker = bge-reranker-v2-m3 or Qwen3-Reranker-0.6B-SEQ-CLS (single forward pass; NEVER the causal form), run on the SAME llama.cpp runtime (llama-cpp-2, vendored cmake build) as embeddings for one native-dep surface. Offer Ettin-32M as an English-only fast-mode reranker, but it CANNOT be the default given the zh-CN/zh-TW i18n contract.

ENGINE CHOICE: llama.cpp (llama-cpp-2) for inference + USearch for the index. Both vendor/compile native C++ during cargo build (rules-compliant); avoid ort/ONNX (pre-1.0, default downloads binaries, CoreML needs from-source build, per-OS dylib codesigning). Both USearch and fastembed-rs are sub-6k-star, so a risk assessment is mandatory regardless; llama-cpp-2 is the cleanest on build rules.

DO NOT use ColBERT/late-interaction: at 140M passages its compressed index is 16-25GB and query latency is hundreds of ms on CPU [ColBERTv2/PLAID]; for a single-user desktop archive it is pure overkill versus quantized single-vector + rescore.

NON-NEGOTIABLE PREREQUISITES the research missed: (a) a resumable, checkpointed, throttled BACKGROUND embedding backfill — embedding 14.4M docs is hours-to-days on 4 cores, so hybrid/vector tools must degrade gracefully to BM25-only while the index fills; (b) a single CPU inference SCHEDULER with priority (interactive rerank > background backfill) so the two don't starve each other or the UI; (c) agent tool descriptions must expose index-readiness so the LLM doesn't call vector-only against an empty index. Expose BM25-only / vector-only / hybrid as separate tools, but gate vector/hybrid on index state and cite real history rows in every result.
**scale_failure:** The research's biggest blind spot: it never did the vector-memory math, and that is where this design dies at 14.4M rows on 8GB RAM.

(1) RAW VECTOR STORAGE IS IMPOSSIBLE AS-DESCRIBED. Qwen3-Embedding-0.6B emits 1024-d float32 = 4096 bytes/vec. 14.4M × 4096 = ~53GB just for vectors, before the HNSW graph (+2-8GB) [zilliz/milvus HNSW overhead refs; web search 2026-06]. On an 8GB box where the canonical SQLCipher DB, the OS, the React/WebView UI, and a loaded ~600MB embedding model already compete for RAM, a 55-61GB in-memory float32 HNSW is off by ~10x. Every candidate in the brief (hnsw_rs, in-RAM HNSW) silently assumes the vectors fit in RAM. They do not. This single omission invalidates the implied "build an HNSW over the embeddings" plan.

(2) THE FIX THE RESEARCH NEVER NAMED: quantization + disk-resident index. Binary quantization (1 bit/dim) = 128 bytes/vec -> 14.4M = ~1.8GB, fits in RAM. int8 (1 byte/dim) = 1024 bytes/vec -> ~14GB (still too big for full RAM but fine memory-mapped on SSD). Standard 2026 production pattern: retrieve with binary (Hamming, SIMD-fast), then rescore top candidates with int8 or float32 -> preserves ~96% of float32 recall at 32x compression [sbert.net embedding-quantization; tacnode 2025; Azure AI quantization docs]. CoRECT (arXiv 2510.19340, 2025-10) confirms non-learned compression holds up "even on up to 100M passages, with statistically insignificant performance loss." Combined with Qwen3's Matryoshka (MRL) truncation (1024->512/256) you get another 2-4x. The research mentioned MRL/quant as model trivia but never connected it to the RAM envelope — it is THE load-bearing decision, not a footnote.

(3) ONE-SHOT BULK IMPORT IS A MULTI-DAY JOB, UNADDRESSED. Embedding 14.4M docs on 4 cores: Qwen3-0.6B does ~6878 prompt tokens/sec single-threaded in llama.cpp, and is reported ~5x SLOWER than bge-m3 (31k tok/s) because it is generation-tuned [llama.cpp issue #19933, 2026]. Real per-document embedding throughput on CPU is cited at 10-50 embeddings/sec [HF Qwen3-Embedding-0.6B discussions]. At even 50 emb/sec across 4 cores (~optimistic 150-200/sec with batching), 14.4M docs = 20-80 hours; at 10-50/sec it is days. The brief treats embedding as a solved precondition. In reality the AI index build is a background, resumable, chunked, days-long job that must survive app restarts and never block the UI — which is exactly PathKeep's "never freeze the main thread" + "rebuildable derived sidecar" contract, but the hybrid-search design has to be built around an index that is FREQUENTLY ABSENT OR PARTIAL. BM25-only must be the always-available default; vector/hybrid degrade gracefully while the embedding backfill catches up.

(4) FTS5 LEXICAL LEG ALSO BREAKS AT SCALE. Confirmed: `ORDER BY rank LIMIT k` on a ~6M-row FTS5 table took 20+s because FTS5 materializes and sorts ALL matches before LIMIT [phiresky/sql.js-httpvfs#10; slingacademy]. At 14.4M rows a broad query ("google") matching millions of rows will hang the BM25 tool unless mitigated: external-content FTS5 (no content duplication), aggressive query gating, and capping candidates. The research flagged this but buried it as a caveat; it is a hard correctness issue for the BM25 agent tool.

(5) RERANKER COST AT SCALE IS FINE — IF the candidate set is capped. Cross-encoders are O(candidates), independent of corpus size, so 14.4M does not change rerank latency; reranking a 50-doc shortlist is the same cost at 14.4M as at 14k. bge-reranker-v2-m3 ~140-200ms, FlashRank tiny ~100ms for 100 docs. The risk is NOT corpus size but (a) running the reranker on every keystroke (must be debounced + post-fusion only), and (b) the autoregressive Qwen3-Reranker causal form at ~380-400ms/query being used by mistake instead of the seq-cls single-forward-pass variant.
