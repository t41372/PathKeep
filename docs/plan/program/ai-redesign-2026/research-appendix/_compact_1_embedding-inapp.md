# In-app embedding inference (Rust, CPU) [embedding-inapp]

## SOTA SUMMARY

As of June 2026 the on-device Rust embedding stack has consolidated around two viable engine families: (a) ONNX Runtime via the `ort` 2.0 crate, used through the higher-level `fastembed` crate, and (b) pure-Rust tensor frameworks, chiefly Hugging Face `candle` (which underpins `text-embeddings-inference`). For PathKeep's no-GPU 4-core/8GB envelope, the default embedding model is locked to the Qwen3-Embedding 0.6B family (Apache-2.0, 0.6B params, 1024-dim, Matryoshka/MRL truncatable 32–1024, 100+ languages, MTEB-multilingual ~64.3), which is the strongest quality-per-MB sub-1B model alongside Google's EmbeddingGemma-300M — but EmbeddingGemma ships under the restrictive custom Gemma Terms of Use (Prohibited Use Policy + downstream flow-down), making Qwen3's Apache-2.0 the cleaner supply-chain default. Quantization is the decisive lever: an int8/uint8 ONNX Qwen3-0.6B is ~0.56–0.62 GB on disk and gives ~99% of fp32 retrieval (NDCG@10 0.689 vs 0.700) while cutting CPU latency ~25%, and uint8-output variants emit 1-byte-per-dim vectors (1 KB/vec at 1024-dim) which is critical because 14.4M fp32×1024 vectors would be ~59 GB whereas uint8 is ~14.7 GB and MRL-truncated 256-dim uint8 is ~3.7 GB. The biggest supply-chain catch is that `ort`'s default `download-binaries` feature fetches a prebuilt ONNX Runtime shared lib over HTTPS at build time (banned pattern), so any ORT path MUST disable default features and either statically link a vendored build or accept load-dynamic. Candle is the cleanest native-dependency story (largely pure Rust, optional MKL/Accelerate, builds with no downloaded C++ binary) and `fastembed` 5.17 now exposes a `qwen3` candle backend, but candle's CPU kernels are generally slower than tuned ONNX Runtime for the same model. The landscape moves fast (fastembed shipped 6 releases in June 2026 alone), so version pinning and a re-eval cadence are warranted.

## CRITIC FINAL REC: Decouple the decision into ENGINE, MODEL TIERS, and INDEX — the research conflated them, and the binding constraints (multi-day embed wall-clock; index RAM > machine RAM) are NOT solved by engine choice.

ENGINE: Adopt candle (via fastembed's `qwen3` candle feature) as the no-native-binary default engine — it is the only path that cleanly passes the supply-chain rules (pure-Rust kernels, 20.5k stars, Apache-2.0/MIT, no downloaded C++). Explicitly REJECT ort for the default path: its download strategy is banned, its compile strategy is the banned build.rs-compiles-C++ pattern, and its system strategy forces you to vendor an ONNX Runtime build (ADR-level commitment). Keep ort as a documented power-user/optimization path only if profiling proves candle is too slow AND someone owns the ONNX Runtime static build. Pin fastembed to an exact version and add a re-eval cadence (it ships ~weekly). Add SHA256 hash-pinning + offline model bundling yourself (fastembed has none) and support a fully-offline first run (ship or sideload the default model, never require a network fetch).
