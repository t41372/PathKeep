# External embedding providers [embedding-external]

## SOTA SUMMARY
As of June 2026 the external embedding ecosystem has effectively standardized on the OpenAI `/v1/embeddings` request/response shape, which is now the lingua franca implemented natively by OpenAI, plus an OpenAI-compatible facade on Ollama (alongside its native `/api/embed`), LM Studio, vLLM, and most cloud vendors. The dominant 2026 capabilities are (a) Matryoshka Representation Learning (MRL) so one model emits a base dimension truncatable to smaller ones with re-normalization (Qwen3 1024→512 loses ~1.4% recall; OpenAI/Gemini/Cohere/Voyage/Jina all support a `dimensions`/`output_dimension`/`output_dimensionality` param), and (b) quantization-aware output dtypes (int8/uint8/binary/ubinary) giving 4x–32x storage cuts with minor recall loss — Voyage and Cohere expose this in-API. The hard operational truth, repeated across 2026 production write-ups (Qdrant, Google Cloud Community, tianpan.co Apr-2026), is that vector spaces are model-specific: changing model, dimension, OR normalization invalidates the index and forces full re-embedding; old vectors cannot be transformed (except experimental "drift-adapter" learned mappings). Best practice is to treat the embedding config as a versioned fingerprint (model id + dimension + dtype + normalization + provider + instruction template), store the source text so re-embedding is always possible, and use blue-green/alias or lazy parallel-index migration. For a local-first app the right design is a single Rust EmbeddingProvider trait with a model-agnostic config (never hardcoded model IDs), the Qwen3-Embedding-0.6B family as the in-app default, and OpenAI-compatible HTTP as the universal external adapter covering Ollama/LM Studio/cloud in one code path. Providers diverge on three things you must abstract: max inputs-per-request (OpenAI 2048 / 300K tokens; Voyage 1000 / 320K tokens; Gemini single-input), whether instructions/task_type are needed (Qwen3, Gemini, Cohere, Jina yes), and whether output is pre-normalized (OpenAI yes; Ollama L2-normalizes; Gemini only at 3072; Qwen3 requires you to normalize) — so PathKeep must always re-normalize defensively after truncation.

## TOP PICKS: OpenAI-compatible /v1/embeddings (universal adapter spec), Qwen3-Embedding family (in-app default — 0.6B), Embedding config fingerprint + alias/lazy migration (re-embed invalidation pattern), Ollama (native /api/embed + OpenAI-compat)

## CRITIC FINAL REC: Build ONE OpenAI-compatible HTTP adapter as the universal external path (covers Ollama native + /v1, LM Studio, TEI, and most cloud in a single thin in-house reqwest+serde client — do NOT add async-openai), plus tiny dedicated adapters only for non-OpenAI-shaped clouds (Gemini task_type/single-input, Voyage/Cohere input_type+output_dtype). Behind that, an EmbeddingProvider trait whose config is a versioned FINGERPRINT = {provider, model_id, effective_dimension, output_dtype, normalized, instruction_template, version} that is PROBED at runtime (never trusted from config — read the actual returned vector length, assert it every batch) and stamped onto the sidecar index; any mismatch = stale = explicit, warned, resumable re-embed with dual-index query during migration. Hard invariants in the trait: carry a query-vs-document role through every call, and L2-renormalize after any MRL truncation.

For the in-app DEFAULT (the part the research under-specified): ship the uint8 Qwen3-Embedding-0.6B ONNX (624 MiB, ~1% loss, 25% faster CPU, native uint8 output) at MRL-512, written into a DISK-RESIDENT memory-mapped quantized sidecar index (USearch view-mode is the leading candidate — vendored C++ build.rs, Apache-2.0, i8/b1 quant, 'serve 100GB on 8GB RAM'; gate it behind a written risk assessment since it's 4.2k stars, with a pure-Rust mmap fallback as plan B). This is the ONLY combination that fits 14.4M in 8 GB: 512-dim uint8 ≈ 0.5 KB/vector → ~7.4 GB of codes that the OS pages from SSD, vs the impossible 54-59 GB of in-memory f32@1024 the research implicitly assumed. NEVER use in-memory full-precision HNSW at this scale, and default to int8/uint8 NOT binary (binary materially hurts Qwen3 and only recovers with full-vector rescoring).

Treat first-embed as a first-class operational subsystem, not a config flag: the one-shot 14.4M embed is a 12-100h CPU-bound job (extrapolated from ~30 ms/doc on a 20-core i9; worse on 4 cores), so it must be chunked, checkpointed/resumable, cancellable, off-main-thread, thermally/battery throttled, with visible progress+ETA — and the app must stay fully usable on deterministic FTS5+trigram+bm25 the entire time AI is off, absent, or mid-build (AI strictly additive, consent-gated, off by default). Cloud embedding providers are incremental/query-time only, NEVER for bulk-archiving the full history (throughput + sovereignty); Gemini's single-input-per-request makes it unfit for bulk by construction. Guard quality with golden-vector + recall@k canaries in `bun run check` so silent instruction/normalization regressions can't ship.
