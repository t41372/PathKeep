# LLM integration (post-litellm 2026) [llm-integration]

## SOTA SUMMARY

As of June 2026 the "litellm router" pattern has dissolved into two layers that serious projects keep separate. Layer 1 is a thin TRANSPORT abstraction: speak the OpenAI-compatible /v1/chat/completions (and increasingly /v1/responses) dialect as the lingua franca, plus native Anthropic Messages and Gemini adapters where features differ; this is exactly what mature Rust crates (rust-genai, async-openai, llm/RLLM, llm-connector, chat-rs) now provide as pure-reqwest clients with no per-provider SDKs. Layer 2 is the AGENT/tooling surface, where Anthropic's Model Context Protocol (MCP) has clearly "won" as the cross-vendor standard for tools/data — OpenAI, Google, Ollama and LM Studio all speak it natively in 2026, and mistral.rs ships both an MCP client and server. OpenAI is steering new work to the Responses API (Assistants API deprecates 2026-08-26; Responses gives a built-in agentic tool loop and 40-80% better cache utilization), but Chat Completions remains universally supported and is still the safest portability baseline; treat Responses as an optional richer adapter, not the floor. For LOCAL inference the field consolidated on llama.cpp's llama-server and Ollama/LM Studio (all OpenAI-compatible, all with JSON-Schema/grammar constrained output and tool calling), with mistral.rs as the strongest pure-Rust embeddable engine (in-process crate OR OpenAI/Anthropic-compatible server, GGUF + ISQ quant, prefix caching, grammar-enforced tools). Structured output converged on JSON-Schema-driven CONSTRAINED DECODING (XGrammar is the server-side default for vLLM/SGLang; Microsoft's Rust-based llguidance underpins OpenAI's Structured Outputs and llama.cpp's -DLLAMA_LLGUIDANCE), so a desktop app should request structured output via the provider's native schema mode and not hand-roll JSON parsing. Net: the modern pattern is "one provider-agnostic Chat-Completions/Responses client trait + optional Anthropic/Gemini native adapters + MCP for tools + a local OpenAI-compatible server (external or in-app)", with model IDs kept entirely in user config. The Rust ecosystem here is real but young — genai/async-openai are well-maintained but small (sub-2k stars), so supply-chain trust hinges on either crossing the 6k bar (rig, mistral.rs, candle do) or accepting a thin, easily-auditable client.

## TOP PICKS: rust-genai (provider-agnostic transport trait, pure-Rust, zero native deps), async-openai (OpenAI/Responses-dialect adapter + Ollama/llama-server reuse), mistral.rs (optional in-app local inference, in-process crate OR OpenAI/Anthropic-compatible server), Model Context Protocol (MCP) (model-agnostic agent tool surface)

## CRITIC FINAL REC: Adopt a TWO-LAYER design, with PathKeep owning a thin internal trait so no third-party type is load-bearing.

LAYER 1 — Transport (own it): define PathKeep's own `LlmProvider` trait (chat, stream, tool-call, structured-output, capability flags). Back it initially with genai (v0.6.x, pinned + vendored + risk-assessment, single-maintainer caveat) because it is the best feature-per-line pure-Rust, zero-native-dep, multi-provider fit and keeps model IDs as plain user-config strings. Keep async-openai available as the OpenAI/Responses adapter if/when Responses-specific features (built-in tool loop, better cache) are wanted. HARD RULE: OpenAI-compatible Chat Completions is the FLOOR only for OpenAI-shaped and LOCAL servers (Ollama, LM Studio, llama-server); Anthropic and Gemini MUST go through NATIVE adapters (genai already provides these) so structured output (strict) and prompt caching actually work — do NOT route Claude through its OpenAI-compat shim. Treat Responses as an optional richer adapter, never the baseline. Wrap API keys in secrecy::SecretString; never serialize them into traces/sidecars.

LAYER 2 — Agent/tools: model PathKeep's history/evidence tools as an IN-PROCESS MCP server using the official rmcp crate (v1.7.0, Anthropic-maintained → trust exemption). This keeps the evidence-citing tool surface model-agnostic and reusable across any provider. External MCP servers = hard default-off, per-server consent (egress boundary). Use constrained/structured output via each provider's NATIVE schema mode, but ALWAYS keep a serde validate-and-repair fallback (constrained = parseable, not semantically correct).

LOCAL INFERENCE policy (the envelope-binding decision): EMBEDDINGS and CHAT are asymmetric. Embeddings (Qwen3-Embedding-0.6B Q8 ≈ 400MB, 20-100ms/chunk CPU) are cheap enough to bundle in-app. CHAT is NOT — default to external providers (Ollama/LM Studio the user already runs; zero supply-chain burden, full sovereignty). If in-app chat ships at all, do it as a llama-server Tauri SIDECAR (not in-process mistral.rs) for OOM isolation and reclaimable RAM, bound to 127.0.0.1, behind: explicit consent + a hard pre-spawn RAM check + a mutex that FORBIDS inference during bulk import. Cap to 0.6B-4B Q4 models. Everything AI is OFF by default; PathKeep stays fully usable with no LLM and no embeddings, per the non-AI deterministic intelligence already in place. Note: the Rust LLM-client landscape is fragmenting, not consolidating (genai 806★ / rig 7.7k★ / mistral.rs 7.3k★ / litellm-rust ~4★) — owning the trait boundary is what insulates PathKeep from that churn.

## CRITIC scale_failure: The LLM-transport layer itself does NOT break at 14.4M rows — it is pure HTTP and stateless; genai/async-openai add no native deps and negligible RAM.
