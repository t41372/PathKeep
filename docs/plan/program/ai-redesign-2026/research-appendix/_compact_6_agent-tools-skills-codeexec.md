# Agent tools, skills & code execution [agent-tools-skills-codeexec]

## SOTA SUMMARY
By mid-2026 three patterns have converged into the standard agent-tooling stack, and PathKeep should adopt all three in layered form. (1) Tools: the round-trip-per-tool-call model is being replaced by "code mode" / "Programmatic Tool Calling" (PTC) — the agent writes a script (TS or Python) that calls your tools inside a sandbox, chains/filters/aggregates locally, and returns only the distilled result. Anthropic's PTC GA'd as `code_execution_20260120` (Jan 2026 version) and reports +11% accuracy with 24% fewer input tokens on BrowseComp/DeepSearchQA; their MCP-as-code post measured 150k→2k tokens (98.7%); Cloudflare's V8-isolate "Code Mode" reports ~99.9% vs naive tool definitions. The companion pattern, Tool Search / progressive tool discovery, GA'd Feb 2026 (~85% token reduction) so a large tool catalog stays out of context until needed. (2) Skills: Anthropic's Agent Skills (SKILL.md, three-level progressive disclosure: name+description always in context, body on activation, bundled files/scripts on demand) became an open standard at agentskills.io on 2025-12-18 and has ~32-40 adopters (Cursor, Codex, Gemini CLI, Copilot, Goose, JetBrains). The 2026 consensus is explicit: skills (procedural knowledge / playbooks, loaded as prompts) and MCP tools (typed connectivity to data/actions) are complementary, not competing — use both. (3) Sandboxing for a desktop app: WASM via Wasmtime is the production answer for running untrusted agent-written code with capability-based isolation (no ambient filesystem/network), deterministic fuel or low-overhead epoch CPU limits (~10% slowdown), per-Store memory limits, and async interruption — all from a single vendored Rust crate (Bytecode Alliance, 18.2k stars, Apache-2.0, v45 in June 2026, OSS-Fuzz). A JS guest can be shipped as a precompiled QuickJS→WASM module (Javy/Bytecode Alliance) and a Python guest as MicroPython→WASM (~362KB, demonstrated June 2026 running under wasmtime with fuel limits). This is a much better supply-chain and isolation fit than embedding V8 (deno_core/rusty_v8) or raw QuickJS C bindings (rquickjs, ~880 stars) directly. For PathKeep specifically the sandbox must be read-only over a narrow, pre-authorized search/query host API (no DB handle, no SQL string, no fs/net), the agent should be able to issue many BM25/vector/hybrid queries and granularity drill-downs in one script turn, and every returned claim must carry the real visit-row IDs that back it.

## CRITIC FINAL REC: Build the agent surface as THREE LAYERS, default-safe and progressively unlocked, with the vector-index decision (not the sandbox) treated as the load-bearing choice.

LAYER 0 - Retrieval floor (ship first, always on, AI-optional-safe): A small set of ~8-12 BOUNDED, read-only structured tools over the canonical DB and sidecars. The non-negotiable one is FTS5+bm25+trigram search (already in the 3.50 stack, incremental, instant, present during long embedding builds and when AI is disabled). Expose the user's requested granularities as explicit tools - raw visits, sessions, query-groups, threads, domains, daily rollups, insights - each returning a BOUNDED top-K with embedded visit_id provenance, plus a cheap fetch_visits(ids) verification tool to satisfy the cite-real-rows constraint. Every tool result carries coverage/freshness metadata so the agent degrades gracefully when the index is partial/absent.

LAYER 1 - Semantic/hybrid (additive, consent-gated): Default index is Qwen3-Embedding-0.6B MRL-truncated to 256-d, BINARY-quantized in RAM (~0.5-1.9GB depending on dim), with a float32/int8 rerank pass over top-K from an mmap'd sidecar - this is the only scheme that fits 14.4M vectors in the 8GB envelope while holding >95% recall. Add a small reranker (Qwen3-Reranker-class) for recall recovery. Plain in-RAM float32 HNSW is OFF THE TABLE at this scale (55-61GB) and must be stated as such. Keep DiskANN/Vamana-on-SSD as a documented fallback IF and only if a high-trust vendored Rust crate exists (else ADR). Embedding/index build runs chunked on the off-main-thread ai worker, resumable, never blocking the UI, and the design assumes it may take hours and be incomplete.

LAYER 2 - Code-mode (OPT-IN, capability-gated upgrade, NOT the default): When the user configures a capable model (local 8B+/Devstral-class or a cloud provider), let the agent write JS that fans out many BM25/vector/hybrid/granularity queries and joins/filters LOCALLY, returning only distilled top-K rows with visit_ids. Engine: Wasmtime (vendored pure-Rust, 18.2k stars, Apache-2.0 - clears the supply-chain gate cleanly) with a precompiled Javy JS guest (build-time .wasm artifact, no shipped C, no rquickjs). Capability-based: ZERO ambient authority; the ONLY host functions are the same bounded read-only query API from Layer 0 - no DB handle, no SQL string, no fs/net. Enforce per-call row caps, per-script total-row and wall-time budgets via epoch_deadline_async_yield_and_update on the worker thread so scripts yield and the UI never freezes. Do NOT ship MicroPython-in-WASM (its own author marks it alpha/risky); offer JS only, or no code-mode on the Python path. On the 0.6B default tier, code-mode is simply unavailable - structured tool-calls are primary there.
