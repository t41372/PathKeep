# Hybrid search + reranking [hybrid-search-rerank]

## SOTA SUMMARY

As of June 2026 the consensus "boring-but-correct" hybrid pipeline is two-to-three stages: (1) run lexical (BM25/FTS5) and dense (ANN) retrieval in parallel, each returning top-K; (2) fuse the two ranked lists; (3) optionally rerank a small shortlist with a cross-encoder. For fusion, Reciprocal Rank Fusion (RRF, k=60) remains the recommended no-tuning default because it is scale-invariant and sidesteps the score-incompatibility of BM25 (unbounded) vs cosine ([-1,1]); peer-reviewed analysis (Pinecone/Bruch et al. "Analysis of Fusion Functions", and the 2508.01405 "Balancing the Blend" study) shows a normalized convex/weighted combination (e.g. min-max + ~0.3 lexical / 0.7 semantic) can beat RRF and is sample-efficient to tune, so the modern stance is "ship RRF, expose an optional tunable weight." Reranking is genuinely additive only when the candidate set is large and noisy (the N>>K rule, ~10:1, e.g. 50->5) and when first-stage MRR@5 is below ~0.85; it is pure latency cost otherwise. On CPU, small cross-encoders are viable: FlashRank MiniLM-L12 reranks 100 docs in ~0.4s and a Tiny model in ~0.1s; bge-reranker-v2-m3 (~568M) is ~140-200ms/query; gte-reranker-modernbert-base (149M) matches a 1.2B reranker on Hit@1; the default-family Qwen3-Reranker-0.6B in its native causal yes/no form is slow (~380-400ms/query, autoregressive) but a seq-cls conversion turns it into a single-forward-pass cross-encoder. Late-interaction/ColBERT is SOTA for quality but its multi-vector index roughly doubles storage (PLAID ~20GiB vs ~13GiB dense at 140M passages) and adds engine complexity, so it is widely judged overkill for a local desktop archive. The under-discussed but decisive local-first detail is engine reuse and supply chain: llama.cpp serves embeddings and reranking from the same runtime (--pooling rank, /v1/rerank), and Rust bindings that vendor C/C++ (llama-cpp-2) compile cleanly during cargo build, whereas the popular ort/ONNX path defaults to downloading prebuilt binaries at build time, which violates strict reproducible-build rules unless reconfigured.

## TOP PICKS: llama.cpp reranking (/v1/rerank, --pooling rank) - shared engine path, Reciprocal Rank Fusion (RRF, k=60) with optional convex/weighted mode, bge-reranker-v2-m3 / Qwen3-Reranker-0.6B-seq-cls as the CPU reranker, Separate agent tools: search_bm25 / search_vector / search_hybrid

## CRITIC FINAL REC: Ship a 3-stage hybrid pipeline, but build it around quantization and a disk-resident index — the parts the original research omitted are the parts that decide whether it works at 14.4M on 8GB.

ARCHITECTURE (CPU, no GPU):

1. LEXICAL leg: keep FTS5 + bm25() as an EXTERNAL-CONTENT table (no content duplication), with hard candidate caps and exact-host/URL short-circuit. This is the always-available default and works with zero AI. Mitigate the `ORDER BY rank LIMIT` blowup (20s at 6M rows) with bounded candidate sets and query gating.
2. DENSE leg: Qwen3-Embedding-0.6B (default, model-agnostic in UI) -> 1024-d, optionally MRL-truncated to 512. Store BINARY-quantized vectors (128 bytes/vec -> ~1.8GB for 14.4M, fits RAM) for first-stage Hamming retrieval, plus int8 vectors on SSD for rescore. Use USEARCH (mmap 'view' index + b1x8/i8 quantization, vendored C++ via cxx-build, Apache-2.0) — write the required risk assessment for its 4.2k-star count. Pipeline: binary ANN -> int8/float32 rescore of top ~200 (this rescore is mandatory for recall AND is most of the quality the research attributed to a separate reranker). This keeps RAM ~2-3GB, not 53GB.
3. FUSION: RRF (k=60) as the principled default — PathKeep has no relevance labels to tune convex weights, so RRF is the right choice, not a compromise. Expose an optional lexical/semantic weight as a power-user knob only.
4. RERANK: OPTIONAL, off by default, on the final fused shortlist (<=50), debounced, never per-keystroke. Default multilingual reranker = bge-reranker-v2-m3 or Qwen3-Reranker-0.6B-SEQ-CLS (single forward pass; NEVER the causal form), run on the SAME llama.cpp runtime (llama-cpp-2, vendored cmake build) as embeddings for one native-dep surface. Ship a verified-correct GGUF with a golden-score test (community conversions are frequently broken). Offer Ettin-32M as an English-only fast-mode reranker, but it CANNOT be the default given the zh-CN/zh-TW i18n contract.

ENGINE CHOICE: llama.cpp (llama-cpp-2) for inference + USearch for the index. Both vendor/compile native C++ during cargo build (rules-compliant); avoid ort/ONNX (pre-1.0, default downloads binaries, CoreML needs from-source build, per-OS dylib codesigning). Both USearch and fastembed-rs are sub-6k-star, so a risk assessment is mandatory regardless; llama-cpp-2 is the cleanest on build rules.

DO NOT use ColBERT/late-interaction: at 140M passages its compressed index is 16-25GB and query latency is hundreds of ms on CPU [ColBERTv2/PLAID]; for a single-user desktop archive it is pure overkill versus quantized single-vector + rescore.

NON-NEGOTIABLE PREREQUISITES the research missed: (a) a resumable, checkpointed, throttled BACKGROUND embedding backfill — embedding 14.4M docs is hours-to-days on 4 cores, so hybrid/vector tools must degrade gracefully to BM25-only while the index fills; (b) a single CPU inference SCHEDULER with priority (interactive rerank > background backfill) so the two don't starve each other or the UI; (c) agent tool descriptions must expose index-readiness so the LLM doesn't call vector-only against an empty index. Expose BM25-only / vector-only / hybrid as separate tools, but gate vector/hybrid on index state and cite real history rows in every result.

## CRITIC scale_failure: The research's biggest blind spot: it never did the vector-memory math, and that is where this design dies at 14.4M rows on 8GB RAM.

Qwen3-Embedding-0.6B emits 1024-d float32 = 4096 bytes/vec. 14.4M × 4096 = ~53GB just for vectors, before the HNSW graph (+2-8GB) [zilliz/milvus HNSW overhead refs; web search 2026-06]. On an 8GB box where the canonical SQLCipher DB, the OS, the React/WebView UI, and a loaded ~600MB embedding model already compete for RAM, a 55-61GB in-memory float32 HNSW is off by ~10x. Every candidate in the brief (hnsw_rs, in-RAM HNSW) silently assumes the vectors fit in RAM. They do not. This single omission invalidates the implied "build an HNSW over the embeddings" plan.
