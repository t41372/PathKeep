# PathKeep 全面審查流水線 — 編排指令

> **Status:** Living document · **Created:** 2026-06-14
> **Scope:** 全代碼庫 + 產品完整度 + UX 審查

---

## 審查哲學

PathKeep 不是一個 spec-driven 的協議庫。它是一個面向真實用戶的桌面產品。
因此本審查流水線有兩條平行的主軸：

1. **代碼主軸** — 代碼是否正確、安全、優雅、高效。
2. **產品主軸** — 用戶體驗是否完整、一致、優秀；功能是否齊全；設計是否忠於願景。

> **文檔是權威參考，但不是法律。**
>
> `docs/` 下的文檔是設計意圖的最佳表述，但它們是活文檔，是 starting point
> 而非 frozen spec。當代碼與文檔不一致時，三種可能性並存：
>
> - **代碼偏離了意圖** → 代碼需要修
> - **文檔落後於演化** → 文檔需要更新
> - **兩者都有道理** → 記錄為 trade-off，交由維護者裁決
>
> 審查子代理必須判斷屬於哪種情況，而不是機械地宣判「代碼違規」。

### 審查定位

這不是一次 PR review（那是增量的、聚焦變更集的）。
這是一次 **里程碑級的全面體檢**，目標是：

- 發現所有重要的技術問題（bug、安全、性能、架構腐化）
- 發現所有重要的產品缺口（該有但沒有的功能、殘缺的用戶旅程）
- 評估 UX 品質（不只是「能用」，而是「好用、優雅、值得信賴」）
- 評估代碼品質（不只是「正確」，而是「乾淨、優雅、可維護」）
- 為下一個里程碑提供優先級指引

## 流水線總覽

```
階段 0  發現與分區          你（編排者）索引代碼庫、文檔、設計規範，劃分審查區域
  │
階段 1  審查子代理          多個子代理平行審查各自的 Focus Area（高召回率目標）
  │                        ├─ 代碼審查子代理（per Focus Area）
  │                        ├─ 產品完整度子代理（跨 Focus Area）
  │                        └─ UX 審查子代理（跨 Focus Area）
  │
階段 2  分診與交叉比對      你（編排者）去重、合併、構建驗證批次
  │
階段 3  對抗性驗證子代理    獨立子代理逐條驗證發現的真實性（高精度目標）
  │
階段 4  綜合與最終報告      你（編排者）合併裁決，識別模式，輸出最終報告
```

階段 1 和階段 3 的子代理之間必須**完全獨立** — 不共享上下文、不繼承推理。
階段 1 優化**召回率**（寧可過報不可漏報），階段 3 優化**精度**（過濾噪聲）。

## 階段 0：發現與分區 (Discovery & Partitioning)

在分派任何子代理之前，先執行以下步驟：

### 0.1 索引代碼庫結構

遍歷項目目錄樹，建立模塊清單與依賴圖。重點關注：

- `src/` 前端模塊邊界（pages / components / lib / app）
- `src-tauri/crates/` 後端 workspace 邊界（vault-core / vault-worker / vault-platform / browser-history-parser）
- `src-tauri/src/` Tauri façade 層（commands / worker_bridge）
- 跨層橋接點（IPC bridge、desktop-contract、backend.ts）

### 0.2 索引文檔體系

通讀以下文檔，建立 **「設計意圖 → 代碼模塊」** 的映射：

| 文檔類別   | 路徑                                  | 作用                                               |
| ---------- | ------------------------------------- | -------------------------------------------------- |
| 願景與需求 | `docs/vision-and-requirements.md`     | 產品定位、核心原則、用戶畫像                       |
| 功能規格   | `docs/features/*.md`                  | 各功能域的詳細需求（活文檔，非 frozen spec）       |
| 設計規範   | `docs/design/*.md`                    | UX 原則、Paper 美學、設計 token、review guardrails |
| 架構文檔   | `docs/architecture/*.md`              | 技術棧、數據模型、模塊邊界、ADR                    |
| 品質矩陣   | `docs/plan/program/quality-matrix.md` | gate 定義                                          |
| 已知問題   | `docs/plan/program/repo-baseline.md`  | 大文件、已知熱點                                   |

### 0.3 識別使命與目標

PathKeep 的使命和目標散佈在多份文檔中，審查前必須建立完整理解：

- **使命**：本地優先、開源、可信賴的瀏覽器歷史長期保存與智能分析
- **三大功能域**：Archive（歸檔）→ Recall（召回）→ Intelligence（洞察）
- **核心原則**：Trust & Transparency、Data Sovereignty、Longevity、Intelligence Is Optional、Recoverability
- **目標用戶**：知識工作者、數據意識強的個人用戶、隱私偏好者
- **設計方向**：Paper + Archival 編輯體美學
- **性能約束**：4 核 3GHz / 8GB RAM，流暢支撐 1440 萬條記錄

### 0.5 標記交叉審查區域

以下區域分配 ≥2 個獨立子代理平行審查，最終交叉比對：

- **IPC 橋接層**（FA-BRIDGE + FA-SHELL）：前端 contract 與後端 command 的一致性
- **搜尋路徑**（FA-EXPLORER + FA-ARCHIVE）：FTS5 查詢、advanced operators、分頁邏輯
- **導入管線**（FA-IMPORT + FA-INGEST）：前端流程 vs Rust 解析/去重邏輯
- **Intelligence 表面**（FA-INTEL + XA-PRODUCT）：功能完整度 vs 實際代碼能力
- **Paper 設計系統**（FA-EXPLORER + XA-UX）：組件實現 vs 設計規範

### 0.6 上下文預算

每個 Focus Area 的代碼 + 相關文檔的總 token 量不應超過子代理上下文的 60%。
如果超出，將 Focus Area 再細分。確保子代理有充足的推理空間。

```
## 審查維度與檢查清單

你必須從以下每個維度逐一審查，不可跳過：

### D1: 功能正確性與完整度 (Correctness & Completeness)
- 代碼行為是否與 features/ 文檔描述一致
- 標記文檔中有描述但代碼未實現的功能（不是 v0.3+ 標記為 deferred 的）
- 標記代碼中存在但文檔未提及的行為（可能是未文檔化的演化，也可能是遺漏）
- 邊界條件是否處理完整（空數據、超長輸入、並發、中斷恢復）
- 錯誤路徑是否都有合理的用戶可見反饋

### D2: 用戶體驗完整度 (UX Completeness)
- 每個用戶可達的狀態是否都有適當的 UI 表現：
  - loading / skeleton / progress
  - empty state（含引導性文案）
  - error state（含診斷信息和下一步建議）
  - disabled / degraded state（含原因說明）
  - success state（含結果確認和後續動作提示）
- PME（Preview → Manual → Execute）流程是否在需要的地方完整實現
- 操作反饋是否即時、明確、誠實（不是 "Processing..." 而是具體在做什麼）
- 可逆性是否對用戶可見（回滾提示、撤銷入口）
- 鍵盤可達性：高風險流程是否可以 keyboard-only 完成
- i18n：所有 user-visible copy 是否在 en / zh-CN / zh-TW 三語都有翻譯

### D3: 設計品質與美學一致性 (Design Quality & Aesthetic Coherence)
- 是否遵循 Paper + Archival 美學方向
- 設計 token 使用是否一致（色票、字體、圓角、間距）
- 字體系統是否正確（Newsreader serif / 系統 sans / JetBrains Mono 各司其職）
- 卡片行為是否遵循 ui-review-guardrails.md（限高、內滾動、全寬白名單）
- 信息密度是否恰當（editorial side：寧可留白，不塞滿）
- 文案是否「講人話」（不用假專業語氣、不空泛、不模糊）

### D4: 架構與工程品質 (Architecture & Engineering Quality)
- 模塊邊界是否遵循 module-boundary-map.md
- 組件職責是否單一（不是上帝組件）
- 業務邏輯是否洩漏進 UI 組件
- Props drilling 是否超過兩層
- 代碼是否優雅、可讀、意圖清晰
- 是否遵守 DRY、SRP、最小驚訝原則
- 是否有不必要的複雜性（過度設計 或 抽象層次不當）
- 異常處理是否完整且語義正確
- 文件行數是否在限制內（新建 ≤ 800、既有 > 1200 需標記、> 1400 僅允許最小改動）

### D5: 性能與規模 (Performance & Scale)
- **心理測試：「1440 萬條記錄時這段代碼會怎樣？」**
- 列表 / 表格是否虛擬化（不允許全量渲染無上限數據集）
- 是否存在主線程阻塞風險
- Rust 側查詢是否有分頁 / 游標（不允許全量加載）
- 內存增長是否有上限
- 是否有 N+1 查詢或不必要的重複計算
- 大量導入場景的吞吐量和回壓策略

### D6: 安全與數據主權 (Security & Data Sovereignty)
- 數據是否有任何機會離開本地（網絡請求、telemetry、外部服務調用）
- 輸入驗證與消毒（特別是用戶提供的搜尋語法、文件路徑）
- SQL 注入防護（FTS5 查詢構建、動態 SQL）
- 路徑遍歷防護（bundle import、export 路徑）
- 加密實現是否正確（SQLCipher、keyring、credential 管理）
- 臨時文件處理和資源清理
- 依賴項的已知漏洞

### D7: 跨模塊一致性 (Cross-Module Consistency)
- 本模塊的約定（命名、錯誤處理模式、i18n key 命名）與項目其他部分是否一致
- Rust ↔ TypeScript 之間的數據格式轉換是否精確（特別是 timestamp、enum variant、nullable 語義）
- 如果發現不一致，在報告中標明並建議統一方向

### D8: 前向兼容與可擴展性 (Forward Compatibility & Extensibility)
- v0.3+ deferred 功能（AI、語義搜尋、MCP、可讀內容）的代碼是否正確標記為 disabled
- 是否有硬編碼假設會在未來演進時成為障礙
- Schema migration 路徑是否安全
- 瀏覽器適配器擴展點是否為新瀏覽器預留空間
```

```
## 項目背景（必讀）

{此處粘貼本提示詞下方的「PathKeep 項目概要」全文}

你正在做的不是代碼審查，而是**產品審查**。你的角色是一位嚴格的
產品經理，從用戶視角全面檢視 PathKeep 是否兌現了它的承諾。

## 你的審查範圍

- **願景文檔**：`docs/vision-and-requirements.md`（完整通讀）
- **功能規格**：`docs/features/*.md`（全部通讀）
- **設計規範**：`docs/design/screens-and-nav.md`、`docs/design/ux-principles.md`
- **智能功能現狀**：`docs/features/intelligence-current-state.md`
- **全部前端代碼**：`src/pages/`、`src/components/`、`src/app/`
- **i18n 目錄**：`src/lib/i18n/catalog/`
- **實際路由表**：`src/app/` 中的路由定義

## 審查維度

### P1: 使命對齊 (Mission Alignment)
PathKeep 承諾三大功能域 — Archive、Recall、Intelligence。
逐一檢視每個功能域：
- 承諾的核心能力是否在代碼中有對應實現？
- 實現的深度是否匹配承諾的期望？（不只是「有」，而是「夠好」）
- 有哪些承諾明確被 defer 到 v0.3+？這些 defer 在 UI 上是否有
  清晰的佔位和說明？

### P2: 功能缺口分析 (Feature Gap Analysis)
從用戶旅程出發，識別「應該有但沒有」的功能：
- **首次使用旅程**：安裝 → 首次備份 → 第一次瀏覽歷史 → 搜尋 → 理解洞察
  這條路徑上有沒有斷點或困惑點？
- **日常使用旅程**：打開 app → 看到有意義的內容 → 找到需要的記錄 → 管理標籤/筆記
  這條路徑上有沒有功能缺失？
- **進階使用旅程**：導入歷史數據 → 管理多瀏覽器 → 導出/遷移 → 安全設置
  這條路徑上有沒有操作死角？
- **錯誤恢復旅程**：操作出錯 → 找到回滾入口 → 理解影響 → 恢復
  這條路徑是否完整？
- **信任建立旅程**：查看審計記錄 → 理解數據完整性 → 驗證備份 → 確認安全
  這條路徑是否讓人安心？

### P3: 用戶旅程完整度 (User Journey Completeness)
對每個主要用戶旅程，檢查：
- 入口是否清晰可見
- 每一步是否有明確的引導
- 成功 / 失敗 / 異常路徑是否都有處理
- 旅程結束後用戶是否知道「接下來做什麼」
- 跨頁面的旅程是否有連貫性（不是到另一頁就丟失上下文）

### P4: 核心原則落地 (Principle Grounding)
逐條檢查 PathKeep 的五大核心原則是否在代碼中有實質落地：
- **Trust & Transparency**：PME 是否在需要的地方都完整實現？
  審計是否真正可追溯？
- **Data Sovereignty**：是否有任何數據離開本地的風險？
  用戶是否真的能直接訪問底層文件？
- **Longevity**：schema migration 是否安全？原始數據是否以超集格式保存？
- **Intelligence Is Optional**：AI 未配置時核心功能是否完全正常？
  AI 表面是否有清晰的 "Coming in v0.3" 佔位？
- **Recoverability**：所有寫入操作是否真正可回滾？
  用戶是否知道去哪裡回滾？

### P5: 競品差異度 (Differentiation Check)
PathKeep 定位為「不是帶 AI 的瀏覽器歷史工具」，而是
「本地、開源、可搜尋、可回顧、可理解的個人瀏覽記憶系統」。
- 產品的獨特價值在代碼中是否有充分體現？
- 有沒有淪為「又一個歷史記錄瀏覽器」而失去差異化的風險？
- Paper + Archival 美學是否成功營造了「個人記憶系統」的感覺？

## 報告格式

### 產品完整度 — 審查報告

- **當前狀態**：完全缺失 / 部分實現 / 有代碼但未串接
- **文檔依據**：哪份文檔暗示或明確要求了這個功能
```

## 階段 3：對抗性驗證 (Adversarial Verification)

> **設計原理**：AI coding agent 審查產品時有五種高頻誤報模式：
>
> 1. **幻影問題** — 代碼實際上是正確的，agent 誤讀了邏輯或遺漏了上下文
> 2. **反向優化** — 提出的「修復」會引入比原代碼更嚴重的問題
> 3. **Trade-off 盲視** — 深思熟慮的工程權衡被當作缺陷報告
> 4. **功能虛構** — 聲稱缺少某功能，但該功能已存在或已被有意 defer
> 5. **品味偏差** — 以個人 UX 偏好取代項目的設計語言和審美方向
>
> 本階段系統性地過濾這五類噪聲。驗證子代理與審查子代理之間必須
> 保持 **完全獨立** — 驗證子代理不繼承審查子代理的推理過程，
> 只接收其結論，然後獨立地試圖推翻它。

```
你是一名 **辯護律師**，你的委託人是當前的代碼實現和產品設計。

你收到的是一組由審查子代理提出的「發現」—— 它們聲稱代碼或產品

最強的反駁理由**，然後基於證據做出公正的裁決。

AI 審查常見的五種誤報模式（你必須主動檢測）：

1. **幻影問題**：代碼實際正確，審查者誤讀了邏輯或遺漏了上下文

#### 逐條裁決

##### 發現 {原始 ID}: {原始標題}

- **原始嚴重等級**: {審查子代理給出的等級}
- **分類**: {BUG / GAP / UX-INCOMPLETE / UX-QUALITY / DESIGN-DRIFT / ...}
- **裁決**: [✅ 確認 / ❌ 駁回 / 🔄 降級 / ⚖️ 確認但屬 trade-off]
- **裁決後等級**: {維持原等級 / 調整後的等級 / 不適用（駁回）}
- **驗證方法**: 用 1-2 句話說明你如何驗證
- **裁決理由**: 用 3-8 句話詳細論證。
  - 如果確認：你獨立驗證後為什麼同意這是真實問題
  - 如果駁回：審查子代理遺漏了什麼上下文或犯了什麼推理錯誤
  - 如果降級：問題存在但嚴重程度被誇大
  - 如果確認但屬 trade-off：原始權衡理由，以及是否值得重新審視
- **如果駁回或降級 — 誤判類型**:
  [幻影問題 / 反向優化 / Trade-off 盲視 / 功能虛構 / 品味偏差 /
   上下文缺失 / 邏輯誤讀 / 嚴重等級誇大 / 修復建議有害 / 其他: {描述}]
- **如果確認 — 對修復建議的評估**:
  原始修復建議是否合理？修復是否會產生級聯影響？
```

### 驗證子代理的關鍵約束

- **絕不自動確認**。每個 ✅ 確認都必須附帶你獨立驗證的具體證據。
- **絕不因為「保守起見」而確認**。無法複現 = ❌ 駁回，不是「以防萬一確認一下」。
- **對 GAP 類發現格外謹慎**。「應該有但沒有」極易受主觀偏差影響。
  必須確認：(a) 願景/功能文檔確實暗示了這個能力，(b) 不是已被有意 defer，
  (c) 真的會影響目標用戶。
- **尊重 PathKeep 的設計語言**。Paper + Archival 美學是有意的選擇。
  「不夠像 Linear」「不夠像 Notion」不是有效的 UX 批評。
  只有偏離了 PathKeep 自己定義的設計方向才是問題。
- **嚴重等級必須校準**。🔴 Critical = 生產環境中的數據損失、安全漏洞、
  或用戶完全無法完成核心功能。「代碼可以寫得更好看」最多是 🔵 Suggestion。
- **讀代碼，不要只讀發現**。驗證子代理的價值在於獨立判斷。

## 階段 4：綜合與最終報告 (Synthesis & Final Report)

### 4.1 匯總驗證結果

- 合併所有驗證子代理的裁決
- 對於同一發現被多個驗證子代理評估的情況，
  取 **更嚴格的裁決**（確認 > 降級 > 駁回）但記錄分歧

### 4.2 模式識別

跨所有報告尋找系統性模式：

- **代碼模式**：反覆出現的同類問題 → 需要項目級修復而非逐點修補
- **產品模式**：反覆出現的功能缺口 → 可能是整個功能域被低估
- **UX 模式**：反覆出現的狀態缺失 → 可能需要組件級的 state 審查
- **性能模式**：反覆出現的規模風險 → 可能需要架構級的分頁策略
- **審查模式**：審查子代理反覆犯的同類誤判 → 記錄用於改進未來的審查

### 4.3 最終報告結構

```markdown
# PathKeep 全面審查報告 — {日期}

1-2 段概述：

- 代碼庫整體健康度
- 產品完整度評估

- 最重要的已確認發現（top 5）
- 驗證階段過濾掉了多大比例的噪聲

## 審查與驗證統計

### 覆蓋範圍

- Focus Areas 數量、審查子代理數量、驗證子代理數量
- 前端覆蓋：{審查了 X / Y 個前端文件}
- 後端覆蓋：{審查了 X / Y 個 Rust 文件}

### 發現存活率

- 階段 1 原始發現數 vs 階段 3 驗證後存活數（附存活率）
- 按分類的存活率分佈
- 按駁回原因分類的誤報統計

- Archive / Recall / Intelligence 完成度評分和論證

### 核心原則落地

按分類分組：

### UX-QUALITY — 用戶體驗品質

## Trade-off 複審清單
```

## 全流程約束

### 文檔立場

- **文檔是權威參考但不是法律**。偏差需要判斷屬性（代碼偏離 / 文檔落後 / trade-off），
  不是機械地宣判違規。
- **ADR 優先**。`docs/architecture/decisions/` 中的 ADR 記錄了經過深思熟慮的權衡，
  除非有強力新證據，不應輕易推翻。
- **引用必須精確**：引用文檔、代碼時，給出具體的文件路徑 / 行號。
  模糊引用 = 無效引用。

### 產品立場

- **功能缺口必須有用戶視角的論證**。「別的產品有所以我們也應該有」不是有效論證。
  必須說明：(a) 哪類目標用戶需要，(b) 在哪個旅程中會感到缺失，(c) 沒有它的後果是什麼。
- **UX 批評必須在項目的設計語言內**。PathKeep 選擇了 Paper + Archival 美學，
  這不是 SaaS 工具的審美標準。批評必須針對項目自己的設計原則，而非通用偏好。
- **v0.3+ 的 deferred 功能不是缺口**。已被明確 defer 的功能（AI runtime、語義搜尋、
  MCP、可讀內容抓取）不應被報告為缺口，除非其 defer 在 UI 上沒有清晰的佔位和說明。

### 工程立場

- **建議必須可操作**：「建議改進」不是有效建議。給出具體方向或代碼草案。
- **尊重項目的性能約束**：4 核 3GHz / 8GB RAM / 1440 萬條記錄。
  優化建議必須在這個約束下可行。
- **尊重項目的品質門檻**：100% 覆蓋率、mutation testing、i18n 三語。
  不要建議降低門檻。

### 階段 1 約束（審查子代理）

- **不要自動降級發現**：不要因為「這可能是有意的」就默認降低嚴重等級。
  標記你的判斷，讓驗證階段來裁決。
- **寧可過報不可漏報**：階段 1 的目標是高召回率。
  假陽性由階段 3 過濾；假陰性沒有補救機會。
- **功能缺口要大膽提出**：如果你覺得「這個產品應該有 X 但沒有」，提出來。
  驗證階段會過濾不合理的。

### 階段 3 約束（驗證子代理）

- **獨立性是生命線**：驗證子代理不得複用審查子代理的上下文。
  從零開始閱讀代碼。
- **裁決必須基於一手證據**：每個 ✅ 和 ❌ 都必須引用你親自閱讀的代碼或文檔。
- **功能缺口的驗證要格外仔細**：搜尋整個代碼庫確認功能真的缺失，
  檢查 BACKLOG.md 確認不是已知已排期，檢查 v0.3 defer 列表。

### 階段 4 約束（綜合）

- **存活率是品質信號**：> 80% 存活率 = 審查精度高。< 40% = 審查提示詞需要校準。
- **駁回發現不可丟棄**：必須進入最終報告的駁回摘要，保持審計可追溯性。
- **Commendation 不可省略**：優秀的設計和實現必須被記錄，不只是找問題。

## 運行注意事項

### Rate limit 處理

長時間審查大概率會撞上 Claude 的 rate limit 導致工作中斷。建議：

- 子代理分批啟動，不要一次全開
- 每個子代理完成後立即保存報告到文件
- 編排者維護進度狀態，中斷後可以從上次完成的位置繼續
- 驗證子代理是最容易中斷的（因為在後半段），確保驗證批次拆分足夠細

### 產出物管理

所有階段的報告保存到 `docs/review/` 目錄下：

- `docs/review/{date}/phase-1/` — 審查子代理報告
- `docs/review/{date}/phase-2/` — 分診結果
- `docs/review/{date}/phase-3/` — 驗證子代理報告
- `docs/review/{date}/phase-4/` — 最終報告

### 觸發時機

建議在以下時機運行全面審查：

- 每個里程碑完成後（里程碑級體檢）
- 重大架構變更後（例如 Paper redesign）
- 準備 release 前
- 累積了大量增量 PR review 後的定期總檢

```
# PathKeep — 項目概要

PathKeep 是一個跨平台 local-first 桌面應用，用於備份、聚合和分析
瀏覽器的歷史記錄。

## 技術棧
- 桌面框架：Tauri 2
- 後端：Rust（workspace：vault-core / vault-worker / vault-platform / browser-history-parser）
- 前端：React 19 + TypeScript + Vite + Tailwind v4
- 數據庫：SQLite / SQLCipher（熱 archive）+ FTS5（全文搜尋）
- AI 框架：rig.rs（可選，v0.3 啟用）
- 原生依賴：vcpkg manifest mode（OpenCC、libsodium via Stronghold）

## 產品定位
本地、開源、可搜尋、可回顧、可理解的個人瀏覽記憶系統。
不是「帶 AI 的瀏覽器歷史工具」。

## 三大功能域
1. **Archive**（歸檔）— 安全、可靠、可審計的瀏覽記錄備份，設計壽命 20 年
2. **Recall**（召回）— 全文搜尋、時間軸瀏覽、篩選、高級搜尋語法、匯出
3. **Intelligence**（洞察）— 確定性洞察已上線，AI/語義搜尋 deferred 到 v0.3

## 核心原則
1. Trust & Transparency — PME 三段式，審計鏈，hash chain
2. Data Sovereignty — 所有數據只存本地，不上雲
3. Longevity — SQLite + 超集 schema + snapshot + migration
4. Intelligence Is Optional — AI 未配置時核心功能完全正常
5. Recoverability — 所有寫入操作可回滾

## 設計方向
Paper + Archival 編輯體美學。Newsreader serif / 系統 sans / JetBrains Mono
三套字體。cream paper 紙感 + 3px 圓角 + slate blue accent。

## 性能約束
目標機 4 核 3GHz / 8GB RAM，流暢支撐 1440 萬條歷史記錄。

## 品質門檻
- 100% JS 覆蓋率（statement / branch / function / line）
- 100% Rust 覆蓋率（line + function）
- i18n 三語覆蓋（en / zh-CN / zh-TW）
- Per-commit mutation testing（desktop contract）
- E2E 測試（browser-preview + desktop-bridge truth）

## 支持的瀏覽器
Chrome、Edge、Firefox、Safari、ChatGPT Atlas、Perplexity Comet

## 當前狀態
v0.2.0 已發布。正在進行 v0.3 Paper Redesign 里程碑。
AI runtime、語義搜尋、MCP、可讀內容抓取已被 defer 到 v0.3 後期。
```
